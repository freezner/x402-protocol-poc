# AGENT.md — 에이전트 운영 규칙

이 문서는 본 PoC 저장소에서 **AI 코딩 에이전트**가 따라야 하는 작업 프로토콜(Operating Protocol)을 정의합니다. `CLAUDE.md` 가 "무엇을 만드는가"를 다룬다면, 이 문서는 "어떻게 작업하는가"를 다룹니다.

---

## 1. 작업 루프 (Work Loop)

모든 작업은 다음 순서를 따른다.

1. **Clarify**: 요구사항이 모호하면 `AskUserQuestion` 류 도구 또는 질문으로 먼저 확인한다.
2. **Plan**: 작업을 단계로 쪼개고 TodoList/Plan 으로 가시화한다. 단계별 검증 항목을 반드시 포함한다.
3. **Read before write**: 파일을 수정하기 전 항상 현재 상태를 읽는다. `specs/`, `docs/` 의 최신 내용을 먼저 확인한다.
4. **Implement**: 작은 단위로 변경한다. 하나의 커밋에는 하나의 관심사(concern)만 담는다.
5. **Verify**: 단순 문법 확인이 아니라 **실제 동작**(예: 엔드포인트 호출, TX 조회)으로 검증한다. 검증 불가 시 그 사실을 사용자에게 명시한다.
6. **Record**: 중요한 결정, 실패한 접근, Latency 측정치 등을 `docs/` 에 기록한다.

## 2. 환경 관리 규칙

- 가상환경은 프로젝트 루트 `./.venv` 에 만든다. 이름/위치를 임의로 바꾸지 않는다.
- 활성화 여부는 `which python` 이 `.venv/bin/python` 을 가리키는지로 확인한다.
- 패키지 추가 시:
  1. `pip install <pkg>` → 2. 즉시 `pip freeze | grep <pkg>` 로 버전 확정 → 3. `requirements.txt` 에 **정렬된 위치**에 추가.
- 개발용 의존성(pytest, black, ruff 등)은 `requirements-dev.txt` 로 분리.

## 3. 시크릿 & 네트워크 안전

- 어떤 경우에도 프라이빗 키·니모닉·API 키를 채팅/파일/로그에 **평문으로 출력하지 않는다**.
- `.env` 는 `.gitignore` 에 포함되어야 한다. 새 변수를 쓸 때마다 `.env.example` 에 **키 이름과 주석만** 추가한다.
- RPC 호출 시 레이트 리밋을 존중한다. 반복 폴링은 최소 간격을 두고 지수 백오프를 사용한다.
- 메인넷 체인 ID(1, 8453 등)에 대한 호출이 발견되면 즉시 중단하고 사용자에게 알린다. 본 PoC는 **테스트넷 전용**이다.

## 4. Django 작업 시 추가 규칙

아직 Django 프로젝트는 존재하지 않으므로, 아래 규칙은 사용자가 "Stage 1 진행" 을 승인한 뒤부터 적용된다.

- 앱(app)은 도메인 단위로 분리한다. 초안: `payments`(결제 인텐트/영수증), `gateway`(402 핸들러·유료 API), `wallet`(지갑 세션 키 관리), `observability`(Latency 로그).
- 모델을 추가하면 **같은 커밋에서** `admin.py` 에 등록한다. 관리자 화면에서 로우 데이터를 조회할 수 없는 모델은 PoC 목적에 어긋난다.
- 마이그레이션은 자동 생성 후 반드시 파일 diff 를 확인한다. `--fake` 사용 금지.
- 타임스탬프 필드는 UTC(`USE_TZ = True`)로 저장한다. Latency 계산의 기준.

## 5. 검증(Verification) 기준

다음은 "완료(done)" 로 선언하기 전 반드시 충족해야 하는 최소 기준이다.

| 단계 | 검증 항목 |
|------|-----------|
| Stage 1 | `python -m venv .venv` → `pip install -r requirements.txt` 가 깨끗하게 끝난다. `.env.example` 이 존재한다. |
| Stage 2 | 임의 API GET 요청 시 `402` 상태 코드와 올바른 `X-402-*` 헤더가 반환된다. 같은 요청에 대해 동일한 `Payment-Id` 가 중복 생성되지 않는다. |
| Stage 3 | 클라이언트 인터셉터가 402를 수신하면 테스트넷에서 실제 TX 를 생성하고 Hash 를 서버로 POST 한다. |
| Stage 4 | 서버가 TX Hash 를 온체인에서 확인한 뒤에만 원래 리소스를 반환한다. Latency 항목이 DB 에 기록된다. |

## 6. 커뮤니케이션 규칙

- 한국어로 응답한다 (사용자가 한국어로 대화 중).
- 결정이 필요한 분기(체인 선택, SDK 선택, 데이터 모델 변경)는 사용자에게 먼저 확인한다.
- 에러/실패는 숨기지 않고 그대로 보고한다. "아마 될 것"이라고 추측하지 않는다.
- 측정 수치는 실측 기반으로만 제시한다. 추정치는 명확히 "추정"이라고 표기한다.

## 7. 금지 사항 요약

- ❌ 사용자 승인 없는 프로젝트 스캐폴딩(`startproject`, `startapp`)
- ❌ 메인넷 키/주소 사용
- ❌ 시크릿의 평문 노출
- ❌ 검증 없는 "완료" 선언
- ❌ `requirements.txt` 와 실제 설치 상태의 불일치
- ❌ 문서 갱신 없는 아키텍처 변경
